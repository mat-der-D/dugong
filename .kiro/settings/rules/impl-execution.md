# Implementation Execution Rules

## Design Deviation Policy

### Principle
design.md は承認済みの設計契約である。実装者が自己判断で設計を変更してはならない。

### Rule
実装中に design.md の仕様と矛盾する制約（言語仕様・ライブラリ制約・コンパイルエラー等）に遭遇した場合:

1. **実装を中断する**
2. **ユーザーに報告する** — 以下を明示すること:
   - design.md のどの仕様と矛盾するか（該当箇所の引用）
   - 矛盾の原因（言語制約・ライブラリ仕様等）
   - 考えられる解決策（複数提示）
3. **ユーザーの判断を待つ** — 承認なしに以下を行わないこと:
   - 可視性（visibility）の変更
   - インターフェース・シグネチャの変更
   - 設計パターンの代替適用
   - 内部構造の変更（フィールドの公開範囲等）

### Rationale
設計逸脱は下流の仕様（他クレート・他モジュール）に波及し得る。技術的に「動く」解決策が設計意図に反する場合がある。判断はユーザーが行う。
